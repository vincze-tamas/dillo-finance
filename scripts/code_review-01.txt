### KR-01 — KATEGORIA kulcs névtévesztés (silent data loss)
**Fájl:** `Config.gs` sor 177, `SheetWriter.gs` sor 114
**Típus:** Névsértés — akkentált vs ASCII kulcsnév
A `Config.gs`-ben az oszlop neve `KATEGORÍA` (U+00CD = latin Í), a `SheetWriter.gs` azonban `c.KATEGORIA` (tiszta ASCII I) formában hivatkozik rá.
```
Config.gs:177      KATEGORÍA:          11,  // K
SheetWriter.gs:114 row[c.KATEGORIA - 1] = kategoria || '';
```
Futás közben `c.KATEGORIA` értéke `undefined`, így `row[undefined - 1]` = `row[NaN]` = `undefined`. A sor 23 elemű tömbben a Kategória (K oszlop) mező soha nem kerül beírásra. Minden beérkező számla kategória mezője üres marad a sheet-ben.
**Javítás:** A `Config.gs`-ben a kulcs neve legyen `KATEGORIA` (ASCII), és minden hivatkozás legyen egységes.

### KR-02 — Cirill karakter a `_validateAllokaciоsSablon_` függvénynévben
**Fájl:** `Validation.gs` sorok 104 és 235
**Típus:** Unicode identifier szennyezés
A függvénynévben a hetedik karakter `о` (U+043E = cirill kis o), nem ASCII `o` (U+006F). Mivel a hívás (`Validation.gs:104`) és a definíció (`Validation.gs:235`) ugyanazt a rontott nevet tartalmazza, jelenleg véletlenül működik. A kockázat abban rejlik, hogy bármilyen jövőbeli szerkesztő vagy keresőeszköz az ASCII változatot hozza létre, ami után két különböző nevű függvény él egymás mellett, és a hívás a cirill névvel silently a nem létező ASCII verziót fogja hívni.
```
Validation.gs:104  _validateAllokaciоsSablon_(sheet, row, value);  // о = U+043E
Validation.gs:235  function _validateAllokaciоsSablon_(...)        // о = U+043E
```
**Javítás:** A nevet cseréljük ki `_validateAllokaciosSablon_`-ra (minden karakter ASCII). Mindkét helyen egyszerre kell átnevezni.

### KR-03 — `afa_szazalek` null-ellenőrzés mindig igaz
**Fájl:** `GeminiOCR.gs` sor 320
**Típus:** Logikai hiba — never-false feltétel
```javascript
afa_szazalek: Number(t.afa_szazalek) !== null ? Number(t.afa_szazalek) : 27,
```
A `Number(...)` mindig number típust ad vissza, amely soha nem egyenlő `null`-lal (`null` nem szám típus). Az egész feltétel mindig igaz, így a `27`-es alapértelmezett soha nem aktiválódik. Ha `t.afa_szazalek` hiányzik (`undefined`), `Number(undefined)` = `NaN`, amelyet a sheet-be ír be.
**Javítás:**
```javascript
afa_szazalek: (t.afa_szazalek !== null && t.afa_szazalek !== undefined) ? Number(t.afa_szazalek) : 27,
```

### KR-04 — MagNet FEJ rekord összeg fillér komment vs tényleges kód ellentmondás
**Fájl:** `BatchGenerator.gs` sor 183
**Típus:** Potenciális formátum hiba (élesítési kockázat)
```javascript
rec += _pad_(String(Math.round(totalAmount)), 15, 'R', '0'); // [47-61] Összeg fillérben (*100)
```
A komment szerint az összeg fillérben szerepel (`*100`), de a kód egyszerűen `Math.round(totalAmount)` — nincs `* 100` szorzás. Ha a MagNet spec valóban filléres formátumot vár (ami a banki batch formátumoknál jellemző), akkor 100-szorosát kellene küldeni. Az TÉTEL rekordon sor 198-ban `// fillér nélkül — ellenőrizd MagNet spec!` komment szerepel, ami jelzi a bizonytalanságot.
**Probléma:** A FEJ és LÁB rekordban a komment azt sugallja, hogy fillér kellene, de a TÉTEL rekordon nincs `*100`. Ha a FEJ-ben kell fillér, akkor FEJ és TÉTEL összegei inkonzisztensek lesznek, a bank visszautasítja a fájlt.
**Javítás előtt:** Kötelező a MagNet CS-ÁTUTALÁS aktuális specifikációjának ellenőrzése. Amíg ez nem történik meg, az élesítés blokkolandó erre a komponensre.

### KR-05 — `_isSameISOWeek_` ISO éven átnyúló hétnél hibás (évhatár bug)
**Fájl:** `WednesdayWorkflow.gs` sorok 189–192
**Típus:** Dátumszámítási hiba
```javascript
function _isSameISOWeek_(a, b) {
  return a.getFullYear() === b.getFullYear() &&
         _getISOWeek_(a)  === _getISOWeek_(b);
}
```
ISO 8601 szerint az ISO év első hete tartalmazhatja az előző naptári év napjait (pl. december 28–31 lehet az 1. ISO hét). Ebben az esetben `a.getFullYear()` = régi év, `b.getFullYear()` = új év, de mindkettő az 1. ISO héten van. A jelenlegi implementáció tévesen különböző hétnek ítéli ezeket, így a `LAST_DIGEST_DATE` ugyanazon a héten futtatott digest esetén nem észleli a duplikátumot.
**Javítás:** Az ISO évet a hét csütörtökje alapján kell meghatározni, nem a naptári évet:
```javascript
function _getISOYear_(date) {
  const d = new Date(date);
  d.setDate(d.getDate() + 3 - ((d.getDay() + 6) % 7));
  return d.getFullYear();
}
function _isSameISOWeek_(a, b) {
  return _getISOYear_(a) === _getISOYear_(b) && _getISOWeek_(a) === _getISOWeek_(b);
}
```

### KP-01 — `PARTNEREK` füleknél hiányoznak a CONFIG oszlopindexek
**Fájl:** `GeminiOCR.gs` sor 455, `WednesdayWorkflow.gs` sor 295, `Validation.gs` sor 91
**Típus:** Magic number — karbantarthatósági és frissítési kockázat
A `CONFIG.COLS` objektumban a `BEJOVO`, `TETEL` és `ALLOKACIO` oszlopok definiálva vannak, de a `PARTNEREK` fülhöz nem. Ennek következtében több helyen mágikus számok szerepelnek:
- `GeminiOCR.gs:455` — `getRange(2, 1, lastRow - 1, 4)` — az oszlop 4 = Kategória hardcode
- `WednesdayWorkflow.gs:295` — `getRange(2, 1, lastRow - 1, 3)` — 3 = Bankszámlaszám hardcode
- `WednesdayWorkflow.gs:298–299` — `row[1]` és `row[2]` indexek hardcode
- `Validation.gs:91` — `if (col !== 8) return;` — 8 = Allokációs sablon hardcode
- `Validation.gs:239, 273, 285` — `sheet.getRange(row, 8)` hardcode
Ha a PARTNEREK fül oszlopstruktúrája megváltozik, legalább 5 helyen kell frissíteni, kézi keresés nélkül.
**Javítás:** `CONFIG.COLS.PARTNER` blokkot kell létrehozni, és minden hivatkozást átvezetni rá.
