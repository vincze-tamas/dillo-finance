## KRITIKUS HIBÁK (élesítési blokkolók)
### KRI-01 — `_getThisWeeksDigestDay_` Vasárnap esetén múlt hétre ugrik
**Fájl:** `WednesdayWorkflow.gs`, sor 119–138
**Típus:** Logikai hiba — helytelen heti navigáció
A `_getThisWeeksDigestDay_` egy vasárnap bemenettel (dow=0) visszalép a megelőző héten lévő szerdára, nem az aktuális hét szerdájára:
```javascript
wednesday.setDate(d.getDate() + (3 - dow)); // 3 - 0 = +3, de vasárnapra ez CSÜTÖRTÖKÖT ad
```
Vasárnap `dow = 0`, tehát `3 - 0 = 3`, a vasárnaptól +3 nap = szerda a KÖVETKEZŐ héten. Ez ISO-szempontból is rossz: a vasárnapot az ISO rendszer az előző héthez sorolja, de a kód a naptári `setDate` logikával a következő hétre ugrik. A trigger minden nap fut, tehát vasárnap is lefuthat (ha pl. kézzel indítják vagy az ütemezés drift-el), és ilyenkor a `_isTodayDigestDay_` hamisan `false`-t ad — a héten soha nem fut le a digest.
**Javítás:** ISO-konform hétfő-alapú heti navigáció:
```javascript
// ISO hétfőre ugrás, majd +2 nap = szerda
const dayOfWeek = d.getDay();
const isoDay    = (dayOfWeek === 0) ? 7 : dayOfWeek; // vasárnap = 7
const wednesday = new Date(d);
wednesday.setDate(d.getDate() - isoDay + 3); // hétfő + 2 = szerda
```

### KRI-02 — `_writeConfigDate_` nincs LockService-szel védve — race condition `LAST_DIGEST_DATE` / `LAST_BATCH_DATE` írásakor
**Fájl:** `WednesdayWorkflow.gs`, sor 245–261
**Típus:** Race condition — dupla digest/batch futás kockázata
A `wednesdayMorningDigest()` és `wednesdayAfternoonBatch()` egymástól függetlenül ellenőrzi az ISO hétalapú deduplikációt, majd igyekszik beírni a `LAST_DIGEST_DATE` / `LAST_BATCH_DATE` értékeket. A `_writeConfigDate_` nem használ LockService-t. Ha a trigger két példányban fut (GAS időnként dupla triggereket indít rövid időn belül), mindkét futás egyszerre olvashatja el az "idén még nem futott" állapotot, és mindkettő lefolytat egy digest- vagy batch-futást.
A gazdasági kockázat az utóbbinál valós: két batch generálódhat egyazon héten, kétszer kerülhet sor a `_updateSSOT_` hívásra, ami a guardrail ellenére is kétszer írja a `KÖTEGEK` fülre az új sort (mert a KÖTEG_ID minden futásnál más lesz a `generateId`-ből).
**Javítás:** A `_writeConfigDate_` és az azt megelőző olvasás/döntés atomi blokkba kell zárni:
```javascript
function wednesdayAfternoonBatch() {
  const lock = acquireLock();
  try {
    const lastBatch = _readConfigDate_('LAST_BATCH_DATE');
    if (lastBatch && _isSameISOWeek_(today, lastBatch)) { return; }
    _writeConfigDate_('LAST_BATCH_DATE', today); // azonnal, lock alatt
  } finally {
    lock.releaseLock();
  }
  // ... batch generálás lock után, már biztonságosan
}
```

### KRI-03 — `_isWorkdaySimple_` rekurzív SpreadsheetApp hivást okoz minden digest-nap-számításnál, ami 6× megsokszorozza az API kvótaterhelést
**Fájl:** `WednesdayWorkflow.gs`, sor 159–162 / `Utils.gs`, sor 70–116
**Típus:** Teljesítmény / GAS kvóta kockázat
`_isWorkdaySimple_` meghívja a `getNextWorkday(date, 0)` függvényt. A `getNextWorkday` minden hívásnál:
1. `SpreadsheetApp.openById()` — egész spreadsheet megnyitása
2. `configSheet.getDataRange().getValues()` — teljes CONFIG fül olvasása — 4×-et (HOLIDAYS aktuális + jövő, WORKING_SATURDAYS aktuális + jövő)
A `_getThisWeeksDigestDay_` egy `for`-ciklusban 3× hívja `_isWorkdaySimple_`-t. Minden trigger-futásnál ez 3 × 5 = 15 Spreadsheet API hívást jelent pusztán a nap ellenőrzésére, mielőtt bármilyen üzleti logika futna. Élesben, ahol a 15 perces Gmail poller is fut és maga is `getNextWorkday`-t hív a fizetési határidők számításakor, az összesített API hívásszám megközelítheti a napi GAS kvótakorlátot (50.000 URL/UrlFetch + Spreadsheet hívás).
**Javítás:** A CONFIG fül adatát egyszer olvassák be egy paraméterként átadott `{holidays, workingSats}` objektumba, és ezt kell átadni a belső `_isWorkday_` hívásoknak. A `_getThisWeeksDigestDay_` paramétert kapjon, ne nyisson Spreadsheet-et.

## KÖZEPES HIBÁK (kockázatos, de nem blokkoló)
### KOZ-01 — `MEGOSZTOTT` kategóriájú számla: `_decideStatusz_` `N/A` összegzőre `BEÉRKEZETT`-et ad vissza, de a logika hibás összefüggés
**Fájl:** `GeminiOCR.gs`, sor 420–421 + `_aggregatePO_`, sor 349–401
**Típus:** Üzleti logika következetlenség
Egy `MEGOSZTOTT` kategóriájú számla esetén `_aggregatePO_` NEM ad `N/A`-t — azt csak az `ÁLLANDÓ` bypass adja. A `MEGOSZTOTT` számlák végigmennek a normál PO aggregáláson, tehát ha a tételeknek nincs PO-juk, `HIÁNYOS` összegzőt kapnak. A `_decideStatusz_` 420–421. sorában az `if (poAgg.poSummary === 'N/A') return 'BEÉRKEZETT'` ág ezért `MEGOSZTOTT` esetén soha nem fut le — a komment ("pl. MEGOSZTOTT + sablon nélkül is elfogadjuk") félrevezető, és a tényleges viselkedés az üzleti szándéktól eltér: egy `MEGOSZTOTT` rezsi-számla PO nélkül `HIÁNYOS_PO` státuszt kap ahelyett, hogy `BEÉRKEZETT` lenne.
**Javítás:** Explicit `MEGOSZTOTT` bypass az `_aggregatePO_`-ban:
```javascript
if (kategoria === CONFIG.KATEGORIAK.ALLANDO ||
    kategoria === CONFIG.KATEGORIAK.MEGOSZTOTT) {
  return { poSummary: 'N/A', poConfidence: 100,
           poReasoning: kategoria + ' kategória — PO ellenőrzés kihagyva' };
}
```

### KOZ-02 — `_loadPendingInvoices_` guardrail fordított logika — JÓVÁHAGYVA + üres KOTEG_ID feltétel
**Fájl:** `WednesdayWorkflow.gs`, sor 294–295
**Típus:** Logikai hiba — megakadályozza az utalt számlák újra-feldolgozását, de fordítva is igaz
```javascript
if (String(row[c.KOTEG_ID - 1] || '').trim() !== '')  return; // ← hard guardrail
```
Ez a guardrail kiszűri azokat a sorokat, amelyek már rendelkeznek `KOTEG_ID`-vel, tehát már szerepeltek egy korábbi kötegben. Ez helyes. Azonban a kondíció csak a KOTEG_ID ürességét ellenőrzi, a `STATUSZ` = `JÓVÁHAGYVA` ellenőrzéssel együtt az implicált feltétel: "JÓVÁHAGYVA státuszú, KOTEG_ID nélküli" sorok kerülnek a kötegbe.
A probléma: ha egy számlán a `STATUSZ` manuálisan visszaállítódik `JÓVÁHAGYVA`-ra (pl. hibás batch után), de a `KOTEG_ID` már ki van töltve, akkor az a számla ki van zárva a következő kötegből — ami helyes viselkedés, de **a hibás batch esetén soha nem kerülhet újra kötegbe**. Nincs erre vonatkozó dokumentáció vagy workflow a kódban. Ez élesben bővebb operatív kockázatot jelent.
**Javítás:** A guardrail kommentet bővíteni, és esetleg egy `KOTEG_ID` törlési procedure dokumentálni:
```javascript
// GUARDRAIL: Ha már van KOTEG_ID, ne szerepeljen újra kötegben.
// Manuális újrafeldolgozáshoz: töröld a V oszlop értékét (autobot@ jogosultság kell).
```

### KOZ-04 — `_buildBankMap_` adószám szanitizálása eltér `_getPartnerKategoria_`-tól — inkonzisztens kulcs-matching
**Fájl:** `WednesdayWorkflow.gs`, sor 324 vs. `GeminiOCR.gs`, sor 458
**Típus:** Adatminőség — bankszámlaszám nem találása potenciális
A `_buildBankMap_` az adószámból kiszűri a nem-szám-és-kötőjel karaktereket:
```javascript
.replace(/[^0-9\-]/g, '')
```
A `_getPartnerKategoria_` viszont csak a számokat tartja meg:
```javascript
.replace(/[^0-9]/g, '')
```
Ha egy partner adószáma a PARTNEREK fülön kötőjelekkel van tárolva (pl. `12345678-2-41`), akkor `_buildBankMap_` kulcsa `12345678-2-41` lesz, de a `_loadPendingInvoices_` a BEJÖVŐ_SZÁMLÁK C oszlopából olvassa az adószámot, amit viszont a GeminiOCR írta be — és a GeminiOCR az OCR-olt adószámot kötőjelek nélkül is tárolhatja (`123456782`). Ez `bankszamla: ''` eredményt ad, és a számla kiesik a batch-ből.
**Javítás:** Egységesítsd az adószám normalizálást mind a két helyen — csak számokat tartsd, a kötőjeleket dobd el mindenhol:
```javascript
.replace(/[^0-9]/g, '') // mindkét helyen
```

### KOZ-06 — `_buildFejRecord_` és `_buildLabRecord_` komment: "forint vagy fillér?" — MagNet formátum nincs megerősítve
**Fájl:** `BatchGenerator.gs`, sor 183, 198–199, 220

### KOZ-08 — `_isProjektszamValid_._cache` és `loadValidProjects._cache` párhuzamos cache — redundáns PROJEKTEK fül olvasás, inkonzisztens értékek lehetségesek
**Fájl:** `Validation.gs`, sor 303–319 vs. `Utils.gs`, sor 305–324
**Típus:** DRY megsértése — két independent cache ugyanarra az adatra
A `loadValidProjects._cache` (Utils.gs) egy `string[]` tömböt cache-el, a `_isProjektszamValid_._cache` (Validation.gs) egy `Set`-et. Mindkettő a PROJEKTEK fül A oszlopát olvassa. Ha a PROJEKTEK fül egy triggerfutás során módosul (Ági hozzáad egy projektet, közben fut az onEdit), a két cache inkonzisztens állapotba kerülhet: `loadValidProjects` a régi listát adja a GeminiOCR-nek, `_isProjektszamValid_` az újat a Validation.gs-nek (vagy fordítva).
**Javítás:** A `_isProjektszamValid_` használja a `loadValidProjects()` cache-ét:
```javascript
function _isProjektszamValid_(projektszam) {
  const projects = loadValidProjects(); // Utils.gs megosztott cache
  return projects.indexOf(String(projektszam).trim()) !== -1;
}
```
A `loadValidProjects._cache = null` cache törlés a `validateAllTetelek()` -ben marad.
