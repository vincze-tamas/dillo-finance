## Review Focus Areas
### 1. The `e.oldValue === e.value` no-op guard (Early exit 2)
**Validation.gs line 37:**
```js
if (e.oldValue !== undefined && String(e.oldValue) === String(e.value)) return;
```
**Status: Important issue.**
The guard has a subtle type-coercion correctness problem and one genuine edge case gap.
**Problem A — `e.value` can be `undefined` when a cell is cleared.**
When a user deletes a cell's content, GAS sets `e.value` to `undefined` (or `''`, depending on the GAS version). `String(undefined)` produces `"undefined"`. If `e.oldValue` is also the string `"undefined"` (unlikely but not impossible for synthetic data), the guard incorrectly exits. More practically: if a user clears a cell that previously had content, `e.oldValue` is e.g. `"JÓVÁHAGYVA"` and `e.value` is `undefined`, so `String(e.value)` is `"undefined"`, the strings differ and the guard correctly passes through — but only by accident. The guard should explicitly handle the `e.value === undefined` case.
**Problem B — Boolean coercion.** A checkbox cell toggled from `true` to `false` gives `e.oldValue = "TRUE"` and `e.value = false` (boolean). `String(false)` is `"false"` (lowercase) while `String("TRUE")` is `"TRUE"`, so they correctly differ. But if a checkbox goes from `false` back to `false` via some programmatic path, both stringify identically. Not a regression — just worth noting the implementation relies on GAS's implicit `String()` coercion being symmetric.
**Recommended fix:**
```js
// Early exit 2: value unchanged (e.g. pressing Enter without editing)
// e.oldValue is undefined when the cell was previously empty — do not skip those.
// e.value is undefined when the cell is cleared — treat as empty string.
if (e.oldValue !== undefined) {
  const oldStr = String(e.oldValue);
  const newStr = (e.value !== undefined && e.value !== null) ? String(e.value) : '';
  if (oldStr === newStr) return;
}
```
This mirrors exactly how `logAudit_` already normalises both values (AuditLog.gs lines 50–55) — aligning the two places to use the same convention.

### 2. `_getRowIdentifier_()` — API calls and performance
**AuditLog.gs lines 108–146.**
**Status: Important issue (performance).**
Every audit log write currently makes **between one and two additional `getRange().getValue()` calls** beyond the ones already executed by the trigger routing code. In the worst case — a SZÁMLA_TÉTELEK edit — it calls `sheet.getRange(row, 1, 1, 2).getValues()` which is one batched read (good). In the BEJÖVŐ_SZÁMLÁK case it calls `sheet.getRange(row, CONFIG.COLS.BEJOVO.SZAMLA_ID).getValue()` — a single-cell read (acceptable but avoidable).
The real concern is the PROJEKTEK branch (lines 128–134): when the edited column is not column 1, it calls `sheet.getRange(row, 1).getValue()` — a live API call on every non-A-column PROJEKTEK edit. Currently PROJEKTEK only watches column A (Validation.gs line 96: `if (col !== 1) return`), so this branch cannot currently be reached through the watched-column filter. However:
- It is a **latent bug**: if someone later adds a second watched column on PROJEKTEK, the code will silently make an extra API read without it being obvious.
- The PARTNEREK branch (line 137) has the same pattern: `sheet.getRange(row, CONFIG.COLS.PARTNER.NEV).getValue()`. `NEV` is column 1. So when column H (ALLOKACIOASSABLON) is edited, it reads column A for the identifier — this is one live API call per PARTNEREK audit log entry. This is real overhead today.
`_getFieldName_` (line 87) also always makes a single-cell `getValue()` call for the header. Since header rows are static and this fires on every watched edit, caching the header in a module-level object (keyed by `sheetName + col`) would save one API call per log entry.
**Recommendation:** Batch the identifier read with the row data read already happening in the routing functions, and pass `rowId` as a parameter to `logAudit_()`, or cache headers at the module level. This avoids extra quota consumption at the cost of a minor refactor.
For now, the implementation is functionally correct and within GAS's 6-minute execution limit for single edits. This becomes relevant at high edit frequency (dozens of concurrent users).

### 3. JOVAHAGYO fix — race condition on empty email
**Triggers.gs lines 331–335:**
```js
const userEmail = Session.getActiveUser().getEmail();
const jovahagyo = userEmail || String(rowData[c.JOVAHAGYO - 1] || '');
if (userEmail) {
  sheet.getRange(row, c.JOVAHAGYO).setValue(userEmail);
}
```
**Status: Correct. One minor observation.**
The fix is correct. The root cause (reading the sheet before writing the email) is resolved. The fallback to the existing sheet value when `userEmail` is empty is appropriate for shared/service account contexts.
One observation: `Session.getActiveUser().getEmail()` returns an empty string (not null/undefined) in installable triggers when the user has not granted the required scope, or in some edge cases where the executing account differs from the editing user. The `|| String(rowData[...] || '')` fallback handles this correctly.
The `if (userEmail)` guard on the `setValue` is correct — it prevents overwriting a pre-existing JOVAHAGYO value with an empty string when the email cannot be determined. This is the right defensive behaviour.
No race condition exists because GAS onEdit triggers are single-threaded within their execution context, and `sheet.getRange().setValue()` is synchronous.

### 4. Audit log call placement relative to KOTEG_ID reversion
**Validation.gs lines 64–67:**
```js
// Audit log ELŐBB — hogy a próbált értéket rögzítsük, nem a visszaállítottat
logAudit_(e, 'KOTEG_ID_OVERWRITE_ATTEMPT');
e.range.setValue(oldVal); // visszaállítás
```
**Status: Correct. The ordering is exactly right.**
Logging before `setValue` ensures the audit record captures `e.value` (the attempted overwrite) and `e.oldValue` (the legitimate value), not the post-reversion state. The comment explains the reasoning. No issue here.
One nuance worth noting: after `e.range.setValue(oldVal)`, the `e` object is not mutated — GAS event objects are immutable snapshots — so `e.value` in the audit record will still be the attempted value even if `logAudit_` were called after. However, calling it before is still the correct and safer convention, and the comment documents the intent clearly.

### 5. GAS-specific issues
**Status: Two issues found.**
**Issue A — `refreshPODropdown_()` called on every PROJEKTEK A-column edit (Important).**
Validation.gs lines 121–123:
```js
_validateProjektszam_(sheet, row, value);
refreshPODropdown_();  // Setup.gs
```
`refreshPODropdown_()` in Setup.gs reads the entire PROJEKTEK A column (`projektek.getRange(2, 1, lastRow - 1, 1).getValues()`), sorts it, then writes a `DataValidation` rule to potentially hundreds of rows in SZÁMLA_TÉTELEK. This is a non-trivial number of API calls (2 reads + 1 write of a range) on every single PROJEKTEK edit, including keystroke-by-keystroke if the user is typing a new project number.
In practice, this is partially mitigated by the no-op guard (Early exit 2) — the trigger only fires when the value actually changes. But every confirmed change (Tab/Enter) on PROJEKTEK will run a full dropdown rebuild. If SZÁMLA_TÉTELEK has many rows, `tetelek.getMaxRows() - 1` could mean setting validation on 999+ rows every time.
**Recommendation:** Consider debouncing by checking whether the sorted project list has actually changed before applying the validation rule:
```js
// Only rebuild if the project list changed
const newList = values.join(',');
// Store previous list in PropertiesService or a script-level cache
```
Alternatively, cap the validation range to `tetelek.getLastRow()` rather than `getMaxRows()`, which would reduce the write scope significantly.
**Issue B — `_setupAuditLog_` datetime format range (Setup.gs, line added after diff line +577).**
```js
sheet.getRange(2, 1, Math.max(sheet.getMaxRows() - 1, 999), 1)
  .setNumberFormat('yyyy-mm-dd hh:mm:ss');
```
`Math.max(sheet.getMaxRows() - 1, 999)` will apply the number format to all existing rows or at least 999 rows. On a freshly created sheet this is fine. On a sheet with many existing rows (e.g. after months of production use), `getMaxRows()` could be large and this format call becomes slow. This is only called from `setupSSOT()` / `_setupAuditLog_()`, not from the trigger path, so it is not a runtime concern — but worth noting for the setup step.
